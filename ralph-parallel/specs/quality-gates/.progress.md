## Learnings
- task-planner.md uses `<mandatory>` blocks for critical rules -- new anti-patterns should follow this pattern
- dispatch-state.json schema: flat JSON with nested `qualityCommands` object -- `baselineSnapshot` follows same nesting pattern
- task-completed-gate.sh uses 5 numbered stages with `# --- Stage N ---` comments -- baseline comparison extends Stage 5
- build-teammate-prompt.py has clean `build_quality_section()` function that accepts quality_commands dict -- easy extension point
- Plugin has no test suite; verification is `bash -n` for shell and `python3 ast.parse` for Python
- Test output format varies wildly across runners; regex cascade with -1 fallback is safest approach
- Explore section in task-planner.md is already in a `<mandatory>` block but uses advisory language ("Spawn...to understand") rather than imperative ("You MUST spawn")
- dispatch.md Steps are numbered sequentially; 4.5 fits cleanly between state write (4) and team creation (5)
- validate-tasks-format.py _extract_task_blocks needed description field added to capture task title text for exemption checks
- COMPILE_ONLY_PATTERNS uses compiled re objects for efficiency; _VERIFY_FIX_MAP provides human-readable suggestions keyed by command prefix
- Verify command extraction handles both inline backtick and multiline code block formats
- capture-baseline.sh uses jq for JSON construction and dispatch-state.json updates; tmp file + mv pattern for atomic writes
- parse_test_count duplicated from task-completed-gate.sh (not extracted to shared lib) to keep scripts standalone
- jq `// empty` returns literal "null" when parent key exists but value is null; must guard with string comparison
- Integer truncation in bash: `1 * 90 / 100` = 0, so small baselines need special threshold logic (BASELINE - 1, min 1)
- ANSI stripping with sed $'s/\x1b\\[[0-9;]*m//g' must use printf '%s' to avoid echo interpretation issues
- _cmds_overlap: prefix matching (startsWith) causes false positives; first-token comparison is correct approach
- Command splitting must handle &&, ||, ;, and | (pipes) -- for pipes, only check first command (the tool)
- N/A case sensitivity: always use .upper() == 'N/A' or re.IGNORECASE for user-facing field values
- validate_verify_commands now returns (errors, warnings) tuple to support empty-verify warnings
