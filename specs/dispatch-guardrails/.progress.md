# dispatch-guardrails

## Original Goal
Prevent dispatch leads from bypassing the prompt generation pipeline when they disagree with auto-partition grouping. Add --manual-groups override to parse-and-partition.py so leads can customize task-to-group assignments while still flowing through build-teammate-prompt.py. Add post-dispatch validation that teammate prompts contain required sections (Signed-off-by, quality commands, file ownership). All fixes must be deterministic script logic, not prose changes.

## Learnings
- `_build_groups_from_predefined()` already strips cross-group dependencies and computes ownedFiles — `_build_groups_from_manual()` should reuse this pattern closely
- build-teammate-prompt.py uses exact section headers: `## File Ownership`, `## Quality Checks`, `## Commit Convention` — validation regex must match these exactly
- validate-tasks-format.py exit code convention: 0=valid, 1=error, 2=format-errors, 3=warnings. validate-prompt.py should use simpler 0=pass, 1=fail since it has a single concern
- Manual groups priority order must be: manual > predefined > auto (most specific wins)
- partition_tasks() already has strategy parameter and content parameter — adding manual_groups follows same extensibility pattern
- Test files use `importlib.util.spec_from_file_location` to load hyphenated filenames — new test file must follow this pattern
- dispatch.md Step 6 generates prompts then spawns via Task tool — prompts must be saved to disk before spawn for Step 6.5 validation
- Fail-open policy is critical: teammates are already spawned when validation runs, killing them wastes compute
- **Design audit (issue #33)**: 10 of 13 CRITICAL findings are design-phase failures (ralph-specum), NOT implementation bugs. Only 2 are implementation deviations that ralph-parallel could theoretically catch.
- Design.md contained 1000+ lines of Metal pseudocode that implementers faithfully followed -- the pseudocode itself had lock-free correctness bugs (ABA, publish fences, inc-on-freed)
- The gpu-os-runtime design.md "Edge Cases" section EXPLICITLY DISMISSED the ABA problem with incorrect reasoning ("slot indices are unique and never reinterpreted") -- this is worse than omission
- Requirements specified functional behavior (AC-6.3: "exactly one zero-crossing") but NOT correctness properties (no "ABA-free", "linearizable", "no use-after-free window")
- Adding design-review as a dispatch gate in ralph-parallel would catch ~15% of issue #33 CRITICALs. The 77% that are design flaws must be fixed in ralph-specum's architect-reviewer phase.
- Lock-free correctness checklist needed in ralph-specum: ABA freedom, publish ordering, no inc-on-freed, cleanup ordering, linearizability
- Research phase extracted PERFORMANCE data from prior art (SlabAlloc 600M/s) but NOT correctness mechanisms (tagged pointers for ABA). Both must be extracted.
- **Concurrency tools research**: Metal/GPU has ZERO automated concurrency analysis tools. No ThreadSanitizer, no race detector, no model checker. Only assertion buffers (manual) work.
- Loom CANNOT model relaxed memory ordering reorderings within a single thread -- would miss the publish-fence bug (Log CRITICAL #1). Only Miri with weak memory emulation catches that.
- Shuttle (AWS) is Loom's practical counterpart -- randomized scheduling, scales to larger tests, catches "most non-adversarial concurrency bugs" within a few hundred iterations. S3 uses it.
- lincheck Rust crate builds on Loom + proptest -- linearizability checking with automatic shrinking. Best practical tool for verifying lock-free DS correctness.
- LLMs score ~76% (GPT-4) on SC concurrency bugs but "frequently fail" on relaxed memory models. Checklist-based prompting converts open-ended review to binary pattern matching, expected to improve reliability significantly.
- The 8-item lock-free correctness checklist maps to ALL 10 design-phase CRITICALs from issue #33. Highest-ROI intervention is prompt injection, not runtime tooling.
- `invalid_atomic_ordering` lint is now deny-by-default in rustc (uplifted from clippy). Catches wrong Ordering args. Zero false positives.
- CDSChecker exhaustively model-checks C11 atomics, found bugs in published concurrent queue implementations. Could theoretically work on translated Metal shader code but no such pipeline exists.
- ConSynergy (2025 paper): hybrid static analysis + LLM achieves 80% precision / 87% recall on concurrency bugs. Not available as drop-in tool.
- **Session isolation research**: `session_id` is a common input field on ALL hooks (confirmed in official docs). Stop hook receives `session_id`, `stop_hook_active`, `last_assistant_message`.
- **CRITICAL**: `session_id` changes on `--resume` (GitHub issues #12235, #8069, #10806). This means coordinator crash + resume = orphaned dispatch. Any session_id-based fix MUST include a fallback mechanism.
- `CLAUDE_ENV_FILE` is ONLY available in SessionStart hooks. Other hook types cannot access it. Plugin SessionStart hooks not receiving it was fixed Dec 2025 (issue #11649) but may still be flaky (issue #15840).
- The current Stop hook never checks `stop_hook_active` -- this is intentionally correct because the hook gives Claude actionable next steps, not just "don't stop."
- Backward compatibility: old `.dispatch-state.json` files without `coordinatorSessionId` should fall through to original behavior (block any session), not silently become invisible.
- Team existence (`~/.claude/teams/{spec}-parallel/config.json`) is the best secondary discriminator when session_id doesn't match. It can't distinguish "unrelated session" from "resumed coordinator" but it narrows the problem.
- Hook timeout is 10s (hooks.json). `jq` on a small JSON file takes ~10ms. Scanning 100+ specs is safe within timeout.
