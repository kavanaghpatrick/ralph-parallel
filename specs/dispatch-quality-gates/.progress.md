# Progress: dispatch-quality-gates

## Original Goal

Fix 6 structural gaps in ralph-parallel dispatch protocol: (1) Pre-dispatch validation of qualityCommands from git root, (2) Pre-merge gate script blocking status="merged" with unchecked tasks, (3) Stop hook must check checkboxes regardless of status, (4) mark-tasks-complete.py must not overwrite unchecked tasks, (5) Coordinator verify failures = hard block in dispatch.md, (6) Baseline failure surfacing. Root cause chain: spec missing cd prefix → baseline exitCode=101 not surfaced → quality gate non-functional → coordinator bypassed stop hook via status="merged" → mark-tasks-complete.py masked failures → tasks silently skipped.

## Learnings

- dispatch-coordinator.sh line 205 `if [ "$STATUS" != "dispatched" ]` is the critical bypass: once coordinator writes "merged", stop hook exits immediately WITHOUT checking if tasks.md checkboxes are all [x]
- The checkbox completion check (lines 229-247) is BELOW the early exit and never runs for status="merged" -- an ordering bug
- task-completed-gate.sh fires for ALL task completions including coordinator-owned ones (no agent-name filter) -- this means VERIFY tasks completed by the coordinator DO go through the gate. The gap is that the gate only runs the task's own verify command, not the full phase gate (build+test+lint)
- mark-tasks-complete.py operates on completedGroups membership, not individual task verification. If a group is in completedGroups, ALL its tasks get marked [x] regardless of whether they were actually run
- capture-baseline.sh writes testCount=-1 when tests fail. In task-completed-gate.sh Stage 5, baseline=-1 causes the ENTIRE test regression check to be skipped for ALL tasks
- The file-ownership-guard.sh pattern (PreToolUse exit 2 blocking) is the strongest enforcement mechanism available -- it blocks at the tool level before the write happens. Same pattern can guard dispatch-state.json status transitions
- TaskCompleted hooks use "exit code only" decision control (not JSON). Exit 2 blocks, stderr becomes feedback. This is already correctly implemented in task-completed-gate.sh
- Stop hooks use top-level `{"decision":"block","reason":"..."}` JSON on stdout + exit 0. dispatch-coordinator.sh uses this correctly
- PreToolUse hooks can return `updatedInput` to modify tool parameters (v2.0.10+) -- potential for transparently adding validation to state writes
- Plugin test suite: 4 Python test files (pytest) + 2 bash test scripts. No lint or typecheck for plugin scripts
- plugin-audit-fixes spec touches same files (task-completed-gate.sh, dispatch-coordinator.sh) -- ordering coordination needed to avoid merge conflicts
- quality-gates v1 and v2 are both complete -- this spec builds on their foundation to close the remaining bypass vectors they didn't address
- Requirements: triple-gate strategy (PreToolUse + stop hook + validate-pre-merge.py) covers both Edit/Write and Bash/jq bypass vectors for status="merged"
- Requirements: strict mode for mark-tasks-complete.py uses checkbox cross-check (cheap) not verify re-run (expensive) -- verify is task-completed-gate.sh's job
- Requirements: VERIFY task detection in task-completed-gate.sh converts 2 prose gaps (advisory phase gate + coordinator verify mandate) into 1 code enforcement via [VERIFY] marker check
- Requirements: stop hook fix is narrow -- only status="merged" falls through to completion check; "aborted"/"superseded"/"stale" still early-exit (backward compat)
- Requirements: hardFail baseline flag distinguishes "tests broken" from "no test command" -- same-exit-code = pre-existing (allow), different-exit-code = new regression (block)
- Requirements: 13 functional requirements map to 7 user stories across 6 gaps -- no gap left uncovered

- Design: merge-guard.sh is a SEPARATE script from file-ownership-guard.sh because they have opposite audiences (coordinator vs teammate). Both register under the same PreToolUse matcher but run in sequence.
- Design: PreToolUse guard uses --skip-quality-commands (checkbox/group check only) because quality commands take 30-60s and PreToolUse must be fast. Full quality enforcement is task-completed-gate.sh VERIFY gate + stop hook.
- Design: Stage 1.5 (VERIFY phase gate) is inserted AFTER Stage 1 (task's own verify), not before it. The task-level verify still fires; VERIFY adds build+test+lint ON TOP.
- Design: Stage 5 hardFail restructure changes the control flow -- test failure is now a branch (hardFail=true + same exit code = allow, different = block) rather than unconditional block. BASELINE_HARD_FAIL and BASELINE_EXIT_CODE must be read BEFORE the test execution block.
- Design: hooks.json supports multiple hooks in the same matcher array -- both file-ownership-guard.sh and merge-guard.sh fire for Write|Edit. First exit-2 blocks the tool call.
- Design: validate-pre-merge.py resolves project root by walking up 2 dirs from dispatch-state.json path (same convention as capture-baseline.sh). No --project-root arg needed.
- Design: mark-tasks-complete.py --strict uses exit code 2 (not 1) for skipped tasks, consistent with the "block" convention used by hooks.
- Design: Bash/jq bypass of status="merged" is accepted risk, mitigated by stop hook as secondary gate. PreToolUse only catches Edit/Write tool calls.
