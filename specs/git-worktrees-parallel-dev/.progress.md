## Original Goal
Research how to run multiple AI coding agent workflows in parallel for faster development, covering git worktrees, multi-agent coordination patterns, Claude Code specifics, coordination challenges, and merge strategies.

## Current Phase
Research - COMPLETE

## Learnings
- Git worktrees are the universal standard for parallel AI agent isolation (adopted by Claude Code, Cursor, Codex, Cline)
- Claude Code has THREE parallelism mechanisms: subagents (Task tool, up to 10), Agent Teams (experimental, full sessions with messaging), and `--worktree` flag (manual)
- Agent Teams are experimental and require `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1` env var
- Clash (Rust CLI) is the leading tool for detecting merge conflicts between worktrees before merge time -- integrates via Claude Code hooks
- Anthropic's own 16-agent C compiler experiment used simple git file locking, no orchestrator -- agents picked "next most obvious problem"
- Cursor found that locking mechanisms degraded 20 agents down to 2-3 effective throughput; three-tier Planner/Worker/Judge pattern worked better
- File-level ownership (each agent owns different files) is the #1 conflict prevention strategy
- Octopus merge (merging multiple branches at once) fails on ANY conflict -- sequential PR-based merging is safer
- Google DORA 2025 data shows quality degradation with AI: +9% bugs, +91% review time, +154% PR size
- METR study found 39-percentage-point perception gap: devs thought they were 20% faster but were actually 19% slower
- Disk space is a real concern: 20min Cursor session with ~2GB codebase used 9.82GB
- Each worktree needs its own environment setup (npm install, env vars, etc.) -- adds overhead
- motlin.com pattern of separate todo.md per worktree with single focused task is effective
- Rate limiting constrains max parallel agents -- spacing with sleep commands helps
- Spec-driven development provides the structured, atomic task definitions needed for effective parallel agent work

### Deep Technical Learnings (2026-02-21)
- Git object writes are content-addressed and use atomic rename -- concurrent object creation across worktrees is inherently safe (idempotent)
- Each worktree has its OWN index file and index.lock at `$GIT_DIR/worktrees/<name>/index` -- no cross-worktree contention on staging operations
- Ref updates use per-ref `.lock` files -- different branches never contend; same-ref updates will fail with "cannot lock ref"
- `git gc` is the PRIMARY concurrency hazard: can delete objects another worktree is using. MUST set `gc.auto 0` for parallel automation
- `git maintenance run` takes an object DB lock preventing concurrent maintenance -- only one can run at a time
- `git merge-tree --write-tree` (Git >= 2.38) performs full merges WITHOUT touching index or working directory -- ideal for pre-merge conflict detection
- merge-tree exit code: 0=clean, 1=conflicts, other=error. Supports `--quiet` for fast boolean check and `--name-only` for conflicted file list
- APFS clonefile (`cp -Rc`) on macOS creates zero-cost instant copies of node_modules -- only diverges when files are modified. Same-volume requirement.
- Symlinked node_modules across worktrees is DANGEROUS -- any `npm install` in one worktree corrupts all others
- `$GIT_DIR` in linked worktree points to private subdir; `$GIT_COMMON_DIR` points to shared .git -- path resolution via `git rev-parse --git-path` handles this automatically
- Integration branch pattern (merge all into temp branch, test, fast-forward main) is safest for automated parallel work
- pnpm content-addressable store provides natural deduplication across worktrees via hardlinks
- Sparse checkout + cone mode can reduce worktree size 50-70% for monorepos but adds O(N*M) pattern matching overhead
- Built-in fsmonitor (`core.useBuiltinFSMonitor`, Git >= 2.37) changes `git status` from O(all files) to O(changed files) -- essential for many worktrees
- Linux inotify default max watches (8192) may be too low for multiple worktrees -- need to increase to 524288+
- lesfurets/git-octopus provides a continuous merge workflow: disposable octopus branch rebuilt on every push, useful for CI on combined feature state
