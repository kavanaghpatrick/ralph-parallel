# Progress: parallel-qa-overhaul

## Original Goal

Overhaul the QA/verification pipeline in ralph-parallel so that parallel-dispatched sub-agents produce actually working, well-tested code. Currently every verify step is static (grep, tsc --noEmit, ls) and catches zero runtime errors. Need: (1) task-planner generates test tasks and meaningful verify commands, (2) teammate prompts include writing tests, (3) per-task gate adds baseline checks beyond verify command, (4) merge checkpoint includes runtime smoke test, (5) verify command quality standards enforced.

## Learnings

- Every verify command in the gpu-metrics-operator spec (25 tasks) is grep/ls/tsc -- zero runtime verification
- The task-planner agent (ralph-specum) has strong E2E principles in its instructions but produces grep-only verify commands in practice -- no examples or enforcement mechanism
- task-completed-gate.sh suppresses all verify output (>/dev/null 2>&1), making failure feedback useless
- build-teammate-prompt.py has no instruction for teammates to write tests or run typecheck after changes
- The gpu-metrics-operator project has NO test runner, NO lint, NO test framework -- only tsc and vite build. Task-planner should have adapted verify commands to use build verification
- [VERIFY] checkpoint tasks are always just tsc --noEmit, never build or runtime check
- dispatch.md lead coordination has no runtime smoke test step at merge checkpoint
- parse-and-partition.py passes verify commands through unchanged -- no quality filtering
- The qa-engineer agent (ralph-specum) already has mock detection and test quality analysis but is never invoked during parallel dispatch
- Related spec parallel-v2 already shipped 11 improvements (1:1 task mapping, stall detection, abort) -- this QA overhaul builds on that foundation
- OpenObserve's 6-phase QA pipeline with Sentinel blocking agent and Healer iteration loop is the best external reference for this problem
- Anthropic's own 16-agent C compiler used the test suite as the primary quality gate -- agents ran full tests and iterated until passing
- Changes to task-planner.md affect the ralph-specum plugin (separate from ralph-parallel) and impact ALL specs, not just parallel ones -- scope decision needed
- 5 modification points identified: task-planner.md (root cause), build-teammate-prompt.py, task-completed-gate.sh, dispatch.md, and optionally parse-and-partition.py
- Scope decision: ralph-specum changes out of scope -- all fixes are compensating controls in ralph-parallel's dispatch layer
- Quality command discovery must support multiple ecosystems (Node, Python, Rust) since ralph-parallel is project-agnostic
- build-teammate-prompt.py's --quality-commands arg keeps the script's interface clean vs reading dispatch state directly
- Per-task gate build frequency (every Nth task) is a pragmatic tradeoff -- running build on every task adds too much latency, never running it misses integration errors
- Verify output suppression (>/dev/null 2>&1) is the single highest-impact bug -- fixing it alone would significantly improve teammate debugging
- Dev server smoke test (start, curl, stop) is the riskiest requirement -- port conflicts, server hang, and cleanup failures. Marked Low priority.
- 6 files modified, 6 user stories, 14 functional requirements. All scoped to ralph-parallel plugin directory.
- Quality command discovery should live in parse-and-partition.py (not a separate script) -- it already reads project files and the discovery result feeds directly into partition output
- Project root can be inferred from --tasks-md path (walk up 2 dirs from specs/$name/tasks.md) -- avoids adding a new required CLI arg
- .dispatch-state.json is the right coordination hub for quality commands between dispatch, gate, and merge -- consistent with existing dispatch state pattern
- --quality-commands as a CLI arg to build-teammate-prompt.py keeps the script stateless (no file I/O for state) -- clean interface boundary
- Verify command classification uses simple substring matching (not regex) against known patterns -- sufficient for short command strings, avoids overengineering
- Concurrent teammates running tsc --noEmit simultaneously is safe (read-only) but may slow each other down
- Build commands that write dist/ are fine -- file-ownership-guard only blocks Write/Edit tool calls, not shell-level writes
- Prompt section ordering: Identity -> Tasks -> File Ownership -> Quality Checks -> Rules (ownership constrains WHAT, quality constrains HOW)
- 4-12 min overhead for full QA pipeline on a 25-task dispatch (20-50% increase) -- justified because current pipeline catches zero runtime errors
- 9 implementation steps, can be parallelized: steps 1-2 (parse-and-partition.py), step 3 (build-teammate-prompt.py), steps 4-5 (gate), step 6 (hooks.json) are all independent file changes
