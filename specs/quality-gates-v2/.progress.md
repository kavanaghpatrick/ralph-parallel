## Learnings
- Stage 6 lint follows exact same structure as Stage 4 build: jq read, interval check, eval, exit 2 on failure. LINT_INTERVAL defaults to 3 (same as BUILD_INTERVAL).
- task-completed-gate.sh Stage 4 (periodic build) is the exact template for Stage 6 (periodic lint) -- same pattern: read cmd from dispatch-state via jq, check interval, run, block on failure
- dispatch-state.json completedGroups contains group names (strings), not task IDs. Must join with groups[].tasks to get task IDs for writeback.
- build-teammate-prompt.py already has a `name` variable in scope at the point where commit convention would be injected -- no new data needed
- The existing quality-gates v1 spec (ralph-parallel/specs/quality-gates/) already implemented baseline capture and verify command validation. This v2 addresses the gaps it missed.
- parse-and-partition.py's partition output includes both group `name` and `tasks` (list of IDs) per group -- mark-tasks-complete.py can rely on this structure in dispatch-state.json
- mark-tasks-complete.py: regex uses re.escape on task IDs and word boundary \b to avoid partial matches (e.g., "1.1" not matching "1.10")
- verify-commit-provenance.py: uses %x00 (null byte) as record separator in git log format to cleanly split multi-line trailer output per commit. Always exits 0 with JSON output even on errors (audit tool pattern).
