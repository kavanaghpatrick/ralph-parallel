## Learnings
- Claude Code conversation is append-only: hook stderr messages injected via exit 2 persist permanently in conversation history. No API to retract/expire them.
- Two signaling approaches for Stop hooks: exit 2 (stderr-only, legacy) vs exit 0 with JSON `{"decision":"block","reason":"..."}` (stdout, recommended). Cannot mix -- exit 2 ignores stdout JSON.
- Plugin bug #10412 (CLOSED): stop hooks with exit 2 fail when installed via plugin system, showing "Stop hook prevented continuation" instead of continuing. Fix status unclear. JSON approach is more reliable for plugins.
- UX issue #12667 (OPEN): JSON approach triggers misleading "Stop hook error:" label in UI. Cosmetic, not functional.
- `stop_hook_active` field: when true, Claude is already continuing from a previous stop hook block. Current code unconditionally exits 0, giving coordinator exactly ONE turn before escape. Official docs say "check this value or process the transcript to prevent running indefinitely."
- Stop hook input schema: `session_id`, `stop_hook_active`, `last_assistant_message`, `transcript_path`, `cwd`, `permission_mode`, `hook_event_name`.
- Auto-reclaim in session-setup.sh can steal dispatches from concurrent sessions. Guard (`TEAM_EXISTS`) is insufficient -- doesn't distinguish "same user restarted" from "different user in parallel."
- jq tmp+mv write pattern is atomic at filesystem level but read-modify-write cycle is not. Low practical risk since Stop hook only reads.
- The sticky stderr problem is unique to Stop hooks. PreToolUse exit 2 blocks a single tool call (transient). TaskCompleted exit 2 prevents task completion (retryable). Stop exit 2 injects a persistent conversation message.
- All 4 hooks in ralph-parallel use exit 2 + stderr for blocking. None use JSON decision control. Migration to JSON is the recommended fix for the Stop hook specifically.
- Hook timeout is 10s (per hooks.json). jq on small JSON takes ~10ms. Scanning 100+ specs is safe within timeout.
- No dedicated test harness for hook scripts. test_session_isolation.sh (from session-isolation spec) is the closest precedent for bash hook testing.
- Quality commands: tsc (build), tsc --noEmit (typecheck), node ./scripts/test.mjs (test). No lint command exists for the project.
- Requirements phase: Whether JSON `reason` persists in conversation history like stderr is UNRESOLVED -- needs empirical testing. This is the single biggest risk to the core fix. If reason persists identically, the fix reduces severity (shorter, state-aware messages) but doesn't eliminate stickiness.
- Requirements phase: Block counter approach (temp file per dispatch+session) is preferred over env var approach because CLAUDE_ENV_FILE is broken on resume (#24775). Temp files survive session restarts.
- Requirements phase: The "all tasks complete" stderr reminder (line 134 of dispatch-coordinator.sh) should be removed entirely -- it contributes to sticky context and the coordinator already has instructions in dispatch.md.
- Requirements phase: Heartbeat approach for cross-session safety is preferred over process-existence checks (platform-specific, fragile) or user-explicit --reclaim (poor UX). 10-minute threshold balances safety vs usability.
- Requirements phase: PreToolUse and TaskCompleted hooks should NOT be migrated to JSON -- their exit-2 usage is transient/retryable and doesn't cause the sticky problem. Only the Stop hook needs migration.
- Requirements phase: Process-unique tmp filenames (file.tmp.$$) prevent a subtle collision when session-setup.sh and dispatch.md write to the same state file near-simultaneously. Current code uses bare `.tmp` suffix.
- Design phase: Block counter format uses `count:status:dispatchedAt` triple to auto-detect dispatch transitions (abort/re-dispatch) without needing explicit cleanup coordination. Simpler than a separate reset mechanism.
- Design phase: Heartbeat write triggers ONLY on block (not every invocation). Allow-exits (terminal status, safety valve) skip heartbeat. This means "heartbeat present" reliably indicates "coordinator was actively blocked" rather than "hook ran at all."
- Design phase: JSON reason messages designed as short pointers (~150-250 chars) referencing current state, not full coordination prompts (~500-800 chars). Even if JSON reason persists in conversation like stderr did, shorter messages degrade more gracefully under compaction.
- Design phase: `block_stop()` should use `jq -nc --arg r "$REASON" '{"decision":"block","reason":$r}'` instead of `printf` to guarantee valid JSON even with unusual spec names containing quotes or backslashes.
- Design phase: macOS date compatibility requires `date -j -f "%Y-%m-%dT%H:%M:%SZ"` (BSD) with fallback to `date -d` (GNU). Both produce epoch seconds for heartbeat age calculation.
- Design phase: The "all tasks complete" stderr reminder (line 134) is removed entirely -- it was the most commonly sticky message since it fires at end-of-dispatch when the model is already trying to stop. Coordinator has dispatch.md instructions for cleanup.
- Design phase: Safety valve (MAX_BLOCKS exit) does NOT delete the counter file. This prevents a loop where: reach MAX_BLOCKS -> allow -> restart -> counter gone -> block again -> reach MAX_BLOCKS -> etc. Counter persists until dispatch transitions to terminal status.
